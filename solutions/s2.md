# Solution S2 — The Obedient Parameter

## The exploit

Authenticate as Alice (`alice-token`, org *Acme*) and call `note_search` with
an explicit `org_id` pointing to Globex:

```json
{
  "name": "note_search",
  "arguments": {
    "token": "alice-token",
    "q": "",
    "org_id": "org_globex"
  }
}
```

In `LAB_S2=vuln` the server returns Globex's notes:

```json
[
  { "id": "n_globex_1", "orgId": "org_globex", "title": "Globex roadmap", ... },
  { "id": "n_globex_2", "orgId": "org_globex", "title": "Globex vendor list", ... }
]
```

Alice, authenticated as Acme, just read another org's private notes.

## Why it works

`note_search` in `src/tools.js` (vuln mode):

```js
guard(async ({ token, q, org_id }) => {
  const session = resolveSession(store, token);
  const effectiveOrgId =
    modes.s2 === "vuln" && org_id ? org_id : session.orgId; // ← trusts caller
  return ok(store.searchNotesByOrg(effectiveOrgId, q));
}),
```

The server resolves who Alice is correctly — `session.orgId = "org_acme"`. But
then it substitutes it with the caller-supplied `org_id` without any membership
check. Authentication answers "who are you?"; this bug skips "are you a member
of the org you are querying?"

## The fix

`LAB_S2=fixed` ignores `org_id` entirely:

```js
const effectiveOrgId = session.orgId; // org_id parameter ignored
```

The parameter stays in the schema (removing it would be a breaking change) but
the server never uses it.

## Key takeaway

This is the "client-supplied scope trusted as authorization" pattern. It is
especially common in search, filter, and list endpoints where a scope parameter
feels like a convenience feature. The fix is: **the org scope must always come
from the server-trusted session, never from the caller's input.**
