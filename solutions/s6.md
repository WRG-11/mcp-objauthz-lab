# Solution S6 — The Uninvited Guest

## The exploit

Authenticate as Alice (`alice-token`, org *Acme*) and call
`note_create_in_org` with a foreign `org_id`:

```json
{
  "name": "note_create_in_org",
  "arguments": {
    "token": "alice-token",
    "org_id": "org_globex",
    "title": "injected",
    "body": "planted by Alice, who is not a Globex member"
  }
}
```

In `LAB_S6=vuln` the server creates the note in Globex's tenant:

```json
{
  "id": "n_101",
  "orgId": "org_globex",
  "ownerId": "u_alice",
  "title": "injected",
  "body": "planted by Alice, who is not a Globex member"
}
```

The next time Bob (Globex) calls `note_list` or `note_search`, Alice's
planted note appears among his own — despite Alice never being a Globex
member.

## Why it works

`note_create_in_org` in `src/tools.js` (vuln mode):

```js
guard(async ({ token, org_id, title, body }) => {
  const session = resolveSession(store, token);
  const targetOrgId =
    modes.s6 === "vuln" && org_id ? org_id : session.orgId; // ← trusts caller
  const note = store.createNote({ orgId: targetOrgId, ownerId: session.userId, title, body });
  return ok(note);
}),
```

The server resolves *who Alice is* correctly — `session.orgId = "org_acme"`.
But it then lets the caller-supplied `org_id` override the write target with
no membership check. This is the write-side twin of S2 (`note_search`'s
scope-as-param bug): instead of *reading* another org's data, the caller
*writes* into it — a data-integrity / poisoning risk rather than a
confidentiality leak.

## The fix

`LAB_S6=fixed` ignores `org_id` entirely as a creation target:

```js
const targetOrgId = session.orgId; // org_id parameter ignored
```

The parameter stays in the schema (removing it would be a breaking API
change, same convention as S2/S4) but the server never honors it as a
destination.

## Key takeaway

This is the "create/update accepting a foreign parent" pattern. Any
mutation endpoint that accepts a parent/org/tenant id as a *write target*
must verify caller membership in that target exactly as strictly as a read
endpoint would — an attacker who can inject data into another tenant does
not need read access at all to cause real damage (poisoned records, spoofed
audit trails, planted "evidence").
