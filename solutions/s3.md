# Solution S3 — The Leaky Batch

## The exploit

Authenticate as Alice (`alice-token`, org *Acme*) and request a batch that
mixes her own note ids with foreign ones:

```json
{
  "name": "note_batch_get",
  "arguments": {
    "token": "alice-token",
    "ids": ["n_acme_1", "n_globex_1", "n_initech_1"]
  }
}
```

In `LAB_S3=vuln` all three notes are returned:

```json
[
  { "id": "n_acme_1",   "orgId": "org_acme",    "title": "Acme acquisition memo" },
  { "id": "n_globex_1", "orgId": "org_globex",  "title": "Globex roadmap" },
  { "id": "n_initech_1","orgId": "org_initech", "title": "Initech TPS reports" }
]
```

Alice reads notes from all three orgs in one call.

## Why it works

`note_batch_get` in `src/tools.js` (vuln mode):

```js
guard(async ({ token, ids }) => {
  const session = resolveSession(store, token);
  const resolved = ids.map((id) => store.getNote(id)).filter(Boolean);
  const result =
    modes.s3 === "vuln"
      ? resolved                                       // ← no org check
      : resolved.filter((n) => n.orgId === session.orgId);
  return ok(result);
}),
```

`note_list` is org-scoped — it only ever returns `session.orgId` notes. So the
ids from other orgs feel "private." But `note_batch_get` resolves each id
directly from the store without confirming it belongs to the session's org.

The ids are also guessable: `n_acme_1`, `n_globex_1`, etc. In a real system
they are UUIDs, but enumerable via timing, error messages, or prior context.

## The fix

`LAB_S3=fixed` filters resolved notes to the session's org before returning:

```js
resolved.filter((n) => n.orgId === session.orgId)
```

Foreign ids are silently dropped. The legitimate batch (own-org ids only) still
works correctly.

## Key takeaway

This is the "list→get asymmetry" pattern. A safe list endpoint creates a false
sense of security about id privacy — and a batch endpoint that trusts those ids
without re-checking org membership breaks it. Every endpoint that resolves
objects by id must independently verify authorization, regardless of whether a
"safer" list endpoint exists alongside it.
