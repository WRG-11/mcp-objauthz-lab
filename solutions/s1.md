# Solution S1 — The Single Outlier

## The exploit

Authenticate as Bob (`bob-token`, org *Globex*) and call `note_delete` with
Acme's note id:

```json
{
  "name": "note_delete",
  "arguments": { "token": "bob-token", "id": "n_acme_1" }
}
```

In `LAB_MODE=vuln` the server responds with:

```json
{ "deleted": "n_acme_1", "title": "Acme acquisition memo" }
```

Bob just deleted a note he has no business touching.

## Why it works

Compare the implementations of `note_get`, `note_update`, and `note_delete`
in `src/tools.js`. All three resolve a note from a client-supplied `id`. The
first two call `requireOrgAccess(session, note)` after the lookup:

```js
// note_get
const note = store.getNote(id);
if (!note) notFound(id);
requireOrgAccess(session, note); // ← present
return ok(note);

// note_delete (vuln)
const note = store.getNote(id);
if (!note) notFound(id);
// requireOrgAccess is MISSING here
store.deleteNote(note.id);
return ok({ deleted: note.id });
```

The single missing line is the entire bug. The server correctly authenticates
*who Bob is* — it just never checks *whether Bob may touch this note*.

## The fix

`LAB_MODE=fixed` adds the missing line:

```js
if (modes.s1 === "fixed") requireOrgAccess(session, note); // ← THE FIX
```

## Two-way confirmation

- **Vuln + cross-tenant** → DELETED ✓ (exploit reproduces)
- **Vuln + `note_get` cross-tenant** → DENIED ✓ (proves single outlier, not global)
- **Fixed + cross-tenant** → DENIED ✓ (fix blocks it)
- **Fixed + same-tenant** → DELETED ✓ (fix does not over-block)

## Key takeaway

This is the "inconsistent authorization — single outlier" pattern. Five tools
are correct; one is not. Automated scanners that test only *whether* a server
has authorization will miss it. You have to read *each* tool individually.
